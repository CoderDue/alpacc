# Wire Protocols of Generated C and CUDA Programs

Programs generated by the C backend (`alpacc c`) and the CUDA backend
(`alpacc cuda`) speak two binary protocols:

- **Batch/test protocol** (default, no flag): every integer is u64
  big-endian, regardless of the grammar's native types. Exists for
  compatibility with `alpacc test generate` and `alpacc test compare`.

- **Server protocol** (`--server`): grammar-native types in host byte
  order (alpacc targets little-endian hosts). Meant for embedding the
  program in other tools: no widening to u64, no byte swapping.

Both protocols exist in all three build modes: lexer-only (`--lexer`),
parser-only (`--parser`), and combined (default).

## Native types

The code generator picks the smallest unsigned integer type that fits each
value domain of the grammar:

- `terminal_t` — token/terminal ids.
- `production_t` — production ids; only defined in builds that contain a
  parser. In combined mode, node ids on the wire also use `production_t`:
  terminal ids always fit in `production_t` (the same invariant the CUDA
  device code relies on for its node-id arrays).
- `index_t` — input positions / lexeme spans; `int64_t` in both backends.

The C and CUDA backends derive these types from the same analysis, so a
given grammar gets identical type sizes in both backends.

Because a client cannot know these sizes in advance, every generated binary
supports:

```
$ ./prog --layout
terminal_t=1
production_t=1
index_t=8
```

`--layout` prints `key=value` lines (sizes in bytes) and exits. Lexer-only
builds omit the `production_t` key (the type does not exist in them); all
other builds print all three keys. Clients should treat unknown keys as
harmless and absent keys as "not used".

## Server protocol (`--server`)

The program reads request frames from stdin (or `-i FILE`) in a loop,
processes each frame independently, writes one response per frame to stdout
(or `-o FILE`), and flushes after each response. EOF on the input ends the
loop. All integers are in **host byte order**.

### Request frame (all modes)

```
u64  frame_len          length in bytes of the content that follows
u64  n                  ┐
payload                 ┘ content (frame_len = 8 + payload bytes)
```

The payload depends on the build mode:

- **lexer-only and combined**: `n` raw input bytes (the text to lex/parse).
- **parser-only**: `n` token ids, each `sizeof(terminal_t)` bytes
  (`frame_len` must equal `8 + n * sizeof(terminal_t)`).

### Response frame

Every response starts with a validity byte:

```
u8   valid              1 = success, 0 = the input was rejected
```

If `valid == 0`, nothing else follows for this frame. If `valid == 1`:

**Lexer-only** — the lexemes:

```
u64          num_lexemes
per lexeme:
  terminal_t token id
  index_t    start (inclusive byte offset)
  index_t    end   (exclusive byte offset)
```

**Parser-only** — the leftmost derivation:

```
u64          num_productions
production_t × num_productions
```

**Combined** — the concrete syntax tree in preorder; parents are node
indices in this order (the root's parent is its own index):

```
u64          num_nodes
per node:
  u8           is_terminal   1 = leaf/terminal node, 0 = production node
  index_t      parent        index of parent node
  production_t id            production id, or terminal id for leaves
  index_t      start         lexeme span (0 for production nodes)
  index_t      end           lexeme span (0 for production nodes)
```

## Batch/test protocol (default mode)

This is the format produced by `alpacc test generate` and checked by
`alpacc test compare`. Every integer is **u64 big-endian**, independent of
the grammar's native types; the generated program widens/narrows at the I/O
boundary.

### Input

```
u64BE  num_tests
per test:
  u64BE  n            bytes (lexer/combined) or token count (parser)
  payload             n raw bytes, or n × u64BE token ids (parser)
```

### Output

One record per test, concatenated. The structure mirrors the server
responses but with all integers as u64 BE:

- **Lexer**: `u8 valid`, then `u64BE num`, then `num` × (`u64BE` terminal,
  `u64BE` start, `u64BE` end).
- **Parser**: `u8 valid`, then `u64BE num`, then `num` × `u64BE` production.
- **Combined**: `u8 valid`, then `u64BE num_nodes`, then per node
  `u8 is_terminal`, `u64BE` parent, `u64BE` id, `u64BE` start, `u64BE` end.

## Backend differences

The two backends implement identical wire formats. The differences are
operational:

- **CUDA lexer-only default input**: without flags it reads a *raw byte
  stream* (not the framed test format) and prints ASCII token spans; pass
  `--inputs` to read a framed `.inputs` batch file. The C backend always
  reads the framed batch format in default mode.
- **CUDA tuning flags**: `--block-size`, `--items-per-thread`,
  `--shared-memory`, `--benchmark`, `--warmup` exist only in CUDA builds.

## Example client

`examples/server_client.py` demonstrates the intended embedding pattern:
query `--layout` once, keep a `--server` process running, and exchange one
frame per request:

```
$ alpacc c grammars/arithmetic.alp -o prog.c && cc -O2 -o prog prog.c
$ python3 examples/server_client.py ./prog both "1 + 2 * x"
```

It works unchanged with C and CUDA binaries in all three build modes
because the frame layout is fully described by `--layout`.

The integration suites (`tests/testc.sh`, `tests/testcuda.sh`) additionally
verify on every random grammar that server-mode responses are byte-identical
to batch-mode output, using `tests/server_test.py` to transcode between the
two protocols.
